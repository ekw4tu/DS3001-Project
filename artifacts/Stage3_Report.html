1. System Overview

Our Stage 2 + Stage 3 pipeline is packaged as a single Python project under src/ (importable library) and scripts/ (CLI entry points). The original Colab notebooks are preserved verbatim in notebooks/; a TA can either open the notebook in Colab or run scripts/evaluate.py to reproduce every metric on the held-out test set with one command.

End-to-end flow

data/Gallery, data/Probe
       │
       ▼
walk_image_tasks ──► InsightFace detect + align ──► ArcFace (512-d) ─┐
       │                        │                                    │
       │                        └─► bbox crop + VGG19 fc2 (4096-d) ──┤
       ▼                                                             ▼
parse_base_identity / parse_condition                        stage2_embeddings.pkl
       │                                                             │
       ▼                                                             ▼
stratified 80/20 split (identity × condition) ──► train_embeddings / test_embeddings
       │
       ├─► Condition LR (ArcFace, VGG19)
       ├─► Identity LR (ArcFace, VGG19)
       ├─► VGG19 L2 + PCA + LR
       ├─► VGG19 MLP head (frozen backbone)
       ├─► VGG19 SupCon contrastive head ──► nearest-centroid (unit sphere)
       └─► ArcFace gallery + identify() cosine ──► gallery_embeddings.pkl

Design choices

  • Shared face crop for apples-to-apples comparison. VGG19 has no face detector; we reuse the InsightFace bounding box so both feature sets see the same pixels. Without this the two extractors would be on different crops and comparisons would be meaningless.
  • Data-driven identity parsing. parse_base_identity(folder, filename) reads canonical labels from a token registry (src/identities.json) with a numeric-stem fallback so classmate folders 1Gallery…38Gallery parse automatically without 38 registry entries.
  • Stratified split on identity × condition. Ensures every (person, condition) cell lands in both train and test, so no bucket is silently untested.
  • Reproducibility via env vars. FR_DATA_ROOT / FR_ARTIFACTS_DIR let the same code run on a laptop, a Mac, or Colab with Drive mounted.

2. Identity and Condition Classification

Rubric §1 · 14% of overall grade · all numbers on the 20% held-out test split (random_state = 42)

2.1 Condition Classification (Logistic Regression, 4%)

A class-balanced logistic regression over each feature set predicts one of five conditions (clean, clean_probe, Expression, Lighting, Occlusion, Side). clean and clean_probe are merged into a single Clean bucket for scoring so the model isn't penalized for splitting on the Gallery/Probe axis.

Feature setOverallCleanExpressionLightingOcclusionSide
ArcFace0.5450.6560.4620.3750.8330.500
VGG190.6120.5380.5000.5590.9170.857

VGG19 wins on condition. This is the opposite of identity (where ArcFace wins — see §2.2). The reason is structural: ArcFace's detection + alignment pipeline deliberately strips out rotation, pose, and lighting so that faces map to canonical 112×112 tiles — those are the exact signals a condition classifier needs. VGG19 receives the raw bounding-box crop and retains orientation + lighting information in its fc2 activations, so an LR on top can recover the condition.

Failure analysis (condition)

  • Lighting vs Clean is the dominant confusion for both models. Many "Lighting" probes are only mildly under-exposed and visually look clean; the LR collapses the boundary. VGG19's 0.559 on Lighting is already a lift over ArcFace's 0.375, consistent with VGG retaining luminance signal.
  • Occlusion and Side are easy (>0.83 and >0.50 on ArcFace, >0.91 on VGG) because sunglasses and profile shots change the global image statistics dramatically.
  • Expression is the hardest middle class — expressions don't shift gross pose or lighting, so the model has only subtle facial-action signal to work with. This is a feature-set limitation, not a tuning issue.

Train vs test gap: the class-balanced LR trains on ~480 samples across 5 classes. Training accuracy saturates near 0.95 while held-out is 0.55–0.61 — a ~35 pt gap. That gap reflects the small-data / high-dimensional regime (512-d or 4096-d features vs ~100 samples per class), not an architectural failure. Adding class_weight='balanced' prevents the LR from defaulting to the majority "clean" class.

2.2 Re-Identification Fine-Tuning on VGG (10%)

The rubric asks us to improve VGG re-ID over the Stage 2 baseline and compare against the easier "LR on top" and the harder "finetune VGG layers" paths. We implemented both, plus a third SupCon contrastive head that directly targets the rubric phrase "clustering / distance-based identification".

ClassifierTypeTest accuracyNotes
ArcFace + LR (baseline comparison)Supervised1.000Perfect on 10-identity closed set
ArcFace cosine identify()Distance-based1.000 (97/97)Production classifier
VGG19 LR raw (Stage 2 baseline)Supervised0.810Rubric baseline
VGG19 L2 + PCA(256) + LRSupervised0.785"LR on top" path from rubric
VGG19 MLP head (frozen backbone)Supervised fine-tune0.8183-layer head + BN + dropout
VGG19 SupCon + nearest-centroidContrastive fine-tuneSee evaluate.py step [7]Our answer to the "clustering / distance" prompt

Why three fine-tuning variants

  • L2 + PCA + LR (0.785). The simplest "LR on top". L2 makes Euclidean distance approximate cosine; PCA-256 keeps 98.4% variance and kills per-feature noise. It does not beat the raw-feature LR because the 10-class regime is already well-separated and dimensionality reduction throws away discriminative dimensions.
  • MLP head (0.818). VGG19 frozen, 3-layer head (4096 → 1024 → 512 → 10) with BN + dropout trained with cross-entropy. Keeps the 140M VGG19 parameters fixed because our dataset is ~600 images — unfreezing VGG19 immediately overfits. The head delivers a modest lift (+0.008) over raw LR; cross-entropy doesn't reshape the embedding geometry itself.
  • SupCon contrastive head. Architecture: VGG19(frozen) → L2 → Linear(4096, 512) → BN → ReLU → Linear(512, 128) → L2. Trained with Supervised Contrastive Loss (Khosla et al. 2020, temperature 0.1). Inference is nearest-centroid cosine on the 128-d unit sphere — exactly the geometry the rubric's "clustering / distance-based identification" language asks for. Cross-entropy only separates classes at the decision boundary; SupCon explicitly pulls same-identity embeddings together and pushes different-identity embeddings apart, which is the property a cosine-NN or k-means lookup exploits.

2.3 Failure Case Analysis

Per-condition identity accuracy (held-out test set)

ConditionArcFaceVGG19 raw LRGap (Arc − VGG)

Where VGG19 fails (and why ArcFace doesn't). The VGG gap is largest on Side (0.286) and Occlusion (0.250) — exactly the conditions that shift face geometry most. Without alignment, VGG19's fc2 activations are dominated by the pose/occlusion signal, so a profile shot of Nico lands closer to any profile shot than to a frontal Nico. ArcFace's pre-embedding alignment warps every face to the same canonical 112×112 template, so pose and occlusion are factored out before the 512-d vector is computed. This is the architectural reason ArcFace reaches 1.000 and the strongest case against the "VGG for identity" approach.
Rubric note on the 75–80% target. The rubric allows for an explanation if performance is hard to achieve, provided we demonstrate progress from the vanilla baseline. Our VGG MLP head (0.818) and L2+PCA+LR (0.785) both meet or exceed the 75–80% range, and the SupCon contrastive head is the principled "distance-based" variant the rubric explicitly asks for. ArcFace at 1.000 confirms the ceiling is not the data but the feature set.

Train vs test

VGG LR trains to ~0.99 and tests at 0.81 — an 18 pt gap. The gap decreases with the MLP head (0.95 / 0.82) because BN + dropout regularize. ArcFace LR trains and tests at 1.0 — the features are already identity-separable, so the LR has nothing left to overfit.

4. In-Class Challenge: Unknown Identity Matching

Rubric §3 · 6% · 4/21 session · 5 unknown feature vectors, 3 correct = full credit

4.1 Input format

The TA distributed 5 .npy files, each a dict-wrapped single vector:

  • ID1.npy, ID2.npy, ID3_UPDATED.npy — 512-d ArcFace (InsightFace buffalo_l)
  • ID4.npy, ID5_UPDATED.npy — 1000-d VGG19 pre-softmax logits (not the 4096-d fc2 we use by default)

Classmates were photographed in-session (1–3 photos each) and added to Drive as 1Gallery/…38Gallery/ folders alongside the 10 original celebrity folders. Total gallery after ingestion: ~180 vectors across 48 identities.

4.2 Route-by-dimensionality workflow

# Sniff dim for each uploaded .npy
import numpy as np
arr = np.load(path, allow_pickle=True)
if arr.dtype == object: arr = arr.item()['features']  # TA's dict wrapper
print(arr.shape)   # (512,) → ArcFace, (1000,) → VGG logits

ArcFace unknowns (512-d) — cosine match against the full gallery via scripts/identify_unknowns.py:

python scripts/identify_unknowns.py /content/ID1.npy --top-k 5
python scripts/identify_unknowns.py /content/ID2.npy --top-k 5
python scripts/identify_unknowns.py /content/ID3_UPDATED.npy --top-k 5

VGG unknowns (1000-d pre-softmax logits) — new scripts/match_vgg_logits.py extracts matching 1000-d logits from the gallery on-the-fly (VGG19 fc2 @ W + b using the predictions layer's own weights) and cosine-matches:

python scripts/match_vgg_logits.py /content/ID4.npy /content/ID5_UPDATED.npy \
    --classmates-only --center --max-sim --gpu --top-k 5

--classmates-only drops the 10 celebrity identities so dense celebrity centroids don't drown out sparse classmate centroids. --center subtracts the gallery-mean logit (removing the common "face-ish" ImageNet component that squeezes every cosine into 0.70–0.85). --max-sim scores each identity by their closest gallery image, not the centroid — preserves the one good shot when classmates have only 1–3 photos.

4.3 Source-level patches merged for the challenge

Three bugs surfaced during class and were fixed permanently:

  • walk_image_tasks previously break-ed after the first of Gallery/ / Gallery / was found, silently dropping the other. Now walks both with dedup via a seen set.
  • parse_base_identity had no entry for numeric folder stems, so 1Gallery…38Gallery all parsed to "Unknown". Added a numeric-stem fallback.
  • identify_unknowns.py couldn't read the TA's object-dtype dict-wrapped .npy files ({'features': array(...)}). Added a branch that unwraps the dict.

Final submission. 5 / 5 predictions produced. The 3 ArcFace matches are high-confidence; the 2 VGG matches are supported by agreement between two independent scoring schemes (centroid on raw logits, and max-similarity on mean-centered logits) — if the features were noise, the two methods would land on different classmates.
Honest caveat on VGG confidence. A k-NN vote over the top-10 nearest gallery images gives only 2/10 agreement for both VGG unknowns — consistent with VGG19's 1000-d ImageNet logits not being face-discriminative at the resolution we need. With 1–3 photos per classmate and 38 classes to discriminate, even a 2/10 vote only barely beats chance (≈1/38 = 2.6/10). The centroid/max-sim agreement is the strongest confidence we can extract from this feature set; better VGG signal would require more photos per classmate, alignment (not just bbox crop), and/or fc2 features instead of pre-softmax logits (all ruled out by the input format the TA provided).
